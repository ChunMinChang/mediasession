Abstract

This specification enables web developers to obtain different levels of platform media focus and implicit access to platform media keys such as hardware keys found on keyboards, headsets, remote controls, and software keys found in notification areas and on lock screens of mobile devices.

1. Introduction

This section is non-normative.

Media is used extensively today on the web to play a variety of different types of content. It is often used to play music, podcasts and radio content. At other times media elements are used to provide background music, sound effects, notification sounds and to render WebRTC media stream content.

When playing media on the web, developers are currently forced to adopt a single default platform modality for playing all media content. On the other hand, native applications can access much richer media integration options with an underlying platform. On mobile devices, native application developers can request many different forms of media integration with the platform to obtain access to headphone buttons, lock screens and notification areas as needed. On desktop devices, native applications have access to keyboard media key events. Native application developers can specify the conditions in which media content should pause or duck on audio interruptions (i.e. pause or lower the volume for the duration of an interruption), continue playing out when application focus is lost or the device screen is switched off and interface with internal and external remote controllers.

This specification provides these platform media integration features for web media. By allowing web developers to express the intended usage of their media content, user agents can supply the most suitable platform-level interactions for that content and enforce logical inter-operation behavior on the current platform between any number of different sources of media.

This specification describes the mechanism that allows web developers to specify the most appropriate category for their media content and for user agents to then use these supplied hints to enforce how different sources of web media should interact with each other, other native media content and the underlying platform.

4. Media sessions

4.1. The MediaSession interface

[Constructor(MediaSessionKind kind)]
interface MediaSession : EventTarget {
  readonly attribute MediaSessionKind kind;

  attribute EventHandler onprevious;
  attribute EventHandler onnext;

  void release();
};

A media session represents one or more media elements within the same top-level browsing context that share the same media category. Each media session then defines the interactions of those media elements with both the underlying platform and other media elements belonging to other media sessions within the user agent.

A media element can have a current media session, which is a media session. When a media element is created the user agent must set its current media session to the current top-level browsing contextâ€™s media session.

The participating media elements of a media session are the media elements whose current media session is that media session. A media sessionâ€™s active participating media elements are the participating media elements of a media session that have a paused attribute of false.

Media sessions have a current state, which describes their current status with respect to any interaction with other media sessions. The possible values are as follows:

IDLE
The media session does not have any platform-level media focus nor is it in a suspended state.
ACTIVE
The media session currently has platform-level media focus.
INTERRUPTED
The media session is currently suspended from having platform-level media focus and its active participating media elements are either paused or ducked based on the media sessionâ€™s media category and pending completion of another media sessionâ€™s media focus.

session = new MediaSession(kind)
Returns a new MediaSession object.

The MediaSession(kind) constructor must create a new MediaSession object with its kind attribute set to kind, and then return it.

When a new media session is created its current state must be initially set to IDLE.

When the media session invocation algorithm is run a media sessionâ€™s current state may be set from IDLE to ACTIVE.

When the media session interruption algorithm is run a media sessionâ€™s current state may be set from ACTIVE to either IDLE or INTERRUPTED depending on its associated media category.

When the media session continuation algorithm is run a media sessionâ€™s current state may be set from INTERRUPTED to ACTIVE.

When the media session release algorithm is run a media sessionâ€™s current state may be set from either ACTIVE or INTERRUPTED to IDLE.

The media session interruption algorithm may cause a media element to either pause or duck depending on its media category. When a media element is to pause the user agent must run its pause() method. When a media element is to duck then the user agent must ensure its output volume is lowered.

The media session continuation algorithm may cause a media element to either unpause or unduck depending on its media category. When a media element is to unduck then the user agent should restore its output volume to the same level that was observed prior to the original corresponding duck request. When a media element is to unpause the user agent must run its play() method.

session . kind
Returns the DOMString value of the intrinsic media category state the given media session represents.

The intrinsic media category state of a media session is the state value of the media category that was set when a media session was created. The possible media categories and their states are defined below.

The kind attribute must return the DOMString value of the intrinsic media category state of the media session.

session . release()
Releases platform-level media focus, sets the media sessionâ€™s current state to IDLE and applies any changes necessary to its active participating media elements.

When the release() method is invoked, the user agent must run the following steps:

  1. Pause each of media sessionâ€™s active participating media elements.
  2. Run the media session release algorithm passing in the media session as the only argument.

The following are the event handlers (and their corresponding event handler event types) that must be supported, as IDL attributes, by all objects implementing the MediaSession interface:

Event handler Event handler event type
onprevious previous
onnext next

When the user presses the MediaPlayPause media key, or an equivalent key, on any hardware or software interface (e.g. on a keyboard or OS-level button) then the user agent must, for each media session known by the user agent, run the following steps:

  1. Let m be the media session.
  2. If m is empty or mâ€™s intrinsic media category state is not Default or Content; or mâ€™s current state is not ACTIVE then abort these steps.
  3. For each active participating media element in m, called active element, run the following substeps:
    1. If active element is paused then unpause active element. Otherwise, pause active element.

When the user presses the MediaTrackPrevious media key, or an equivalent key, on any hardware or software interface (e.g. on a keyboard or OS-level button) then the user agent must, for each Content-based media session that is currently ACTIVE, queue a task to fire a simple event named previous at its MediaSession object.

When the user presses the MediaTrackNext media key, or an equivalent key, on any hardware or software interface (e.g. on a keyboard or OS-level button) then the user agent must, for each Content-based media session that is currently ACTIVE, queue a task to fire a simple event named next at its MediaSession object.

5. Media categories

A media category is a keyword that maps to a well-defined interaction state for a media element that user agents use to enforce platform-specific interaction and interoperation between web media.

enum MediaSessionKind {
  "content",
  "transient",
  "transient-solo",
  "ambient"
};

The following table lists the available media categories keywords and states â€” the keywords in the left column map to the states in the cell in the second column on the same row as the keyword. The attribute should be set based on the preferred definitions in the tableâ€™s third and fourth columns.

Category State Definition Examples
"content" Content
  • Indefinitely pauses all other active Default and Content content when playback begins.
  • Displays lock-screen and notification area user interfaces when playback begins.
  • Reacts to changes in both hardware and software-based media key buttons when playback begins.
  • Ducks itself when Transient content starts interrupting. Unducks itself when Transient content is paused or ends playback.
  • Pauses itself when Transient Solo content starts interrupting. Unpauses itself when Transient Solo content is paused or ends playback.
  • Continues playing normally when Ambient content starts interrupting.
Music, podcasts, radio streams.
"transient" Transient
  • Ducks all other active Default and Content content when playback begins. Unducks all other interrupted Default and Content content when paused or playback ends.
  • Does not display lock-screen and notification area user interfaces when playback begins.
  • Does not react to changes in both hardware and software-based media key buttons when playback begins.
  • Pauses itself when Transient Solo content starts interrupting. Unpauses itself when Transient Solo content is paused or ends playback.
  • Continues playing normally when other Transient or Ambient content starts interrupting.
Notification sounds. Spoken driving directions where playback of music in the background is acceptable.
"transient-solo" Transient Solo
  • Pauses all other active Default, Content, Transient and Transient Solo content when playback begins. Unpauses all interrupted Default, Content, Transient and Transient Solo content when paused or playback ends.
  • Does not display lock-screen and notification area user interfaces when playback begins.
  • Does not react to changes in both hardware and software-based media key buttons when playback begins.
  • Pauses itself when other Transient Solo content starts interrupting. Unpauses itself when other Transient Solo content is paused or ends playback.
  • Continues playing normally when Ambient content starts interrupting.
Playback of driving directions. Other spoken notifications.
"ambient" Ambient
  • Does not interact with any other Default, Content, Transient, Transient Solo or Ambient content during playback.
  • Does not display lock-screen and notification area user interfaces when playback begins.
  • Does not react to changes in both hardware and software-based media key buttons when playback begins.
  • Pauses itself when its top-level browsing context is not visible (i.e. the page is placed in the background or the user agent is minimized or the screen is switched off). Unpauses itself when its top-level browsing context is at least partially visible again (i.e. the user agent is in the foreground and the page is focused again).
UI sounds. In-game sound effects.
"" (empty string) Default
  • No explicit kind, or the kind of the media given is not recognized by the user agent.
Legacy media content

6. Assigning a media session to media elements

6.1. Extensions to the HTMLMediaElement interface

partial interface HTMLMediaElement {
  // media session
  attribute DOMString kind;
  attribute MediaSession? session;
};
media . kind
Returns the category the given media falls into. The possible media categories are defined above.
media . session [ = session ]
Returns the current media session for the media element, if any; returns null otherwise.

Can be set, to set an explicit MediaSession. Doing so updates the media elementâ€™s kind attribute.

The kind IDL attribute is intended to provide a hint to the user agent about what the author thinks will lead to the best user experience for media playback. The attribute may be ignored altogether, for example based on explicit user preferences or based on the capabilities of the underlying platform.

The kind attribute must reflect the video and audio element content attribute of the same name, limited to only known values.

When a media elementâ€™s kind attribute is set, changed, or removed then the user agent must run the update the attached media session algorithm, passing in media element as the only argument.

A media elementâ€™s kind attribute can be changed at any time even once the media resource is being played; but it is only read by the user agent at the time that it is set, changed or removed. Thus, setting, changing or removing this attribute once the media element is potentially playing does not dynamically change the media session to which the media element belongs until the media session invocation algorithm is subsequently re-run.

The session attribute on a media element, on getting, must return the elementâ€™s current media session, if any, or null otherwise. On setting, the user agent must run the following steps:

  1. Let m be the media element in question.
  2. Let old media session be mâ€™s current media session, if it has one, and null otherwise.
  3. Let new media session be null.
  4. Let m have no current media session, if it currently has one.
  5. If the new value is null then set the elementâ€™s kind attribute to the empty string. Otherwise, set elementâ€™s kind attribute to the value of current media sessionâ€™s kind attribute.
  6. Let mâ€™s current media session be the new value or the top-level browsing contextâ€™s media session if the new value is null.
  7. Let new media session be mâ€™s current media session.
  8. Update media sessions: If old media session and new media session are the same (whether both null or both the same media session) then abort these steps.
  9. If m is an active participating media element of old media session then pause m.
  10. Remove m from old media sessionâ€™s participating media elements.
  11. If old media session is not null and no longer has one or more active participating media elements, then run the media session release algorithm, passing in old media session as the only argument.
  12. Add m to new media sessionâ€™s participating media elements.

6.2. Assigning a media session declaratively

The kind content attribute on media elements is an enumerated attribute and must return the media elementâ€™s media category, if it has one, or the empty string otherwise.

When a media element is created with a kind content attribute then the user agent must run the update the attached media session algorithm, passing in media element as the only argument.

The update the attached media session algorithm takes one argument, current media element, and consists of running the following steps:

  1. Let media category state be the current media elementâ€™s kind attributeâ€™s state value as defined in the list of possible media categories. If no known category match is found, let media category state be Default.
  2. If media category state is Default then let media session be the current top-level browsing contextâ€™s media session and set current media elementâ€™s current media session to the current top-level browsing contextâ€™s media session. Otherwise, if current media elementâ€™s intrinsic media category state is the same as media category state then let media session be current media elementâ€™s current media session object. Otherwise, let media session be a new MediaSession object and set media sessionâ€™s intrinsic media category state to the value of media category state.
  3. If media category state is Default then set current media elementâ€™s session attribute to null, abort the remaining steps and return.

    Default media categories do not expose a session attribute but still maintain an internal current media session reference.

7. Media session invocation

When the play() method on a media element is invoked and the media elementâ€™s paused attribute is true and the media elementâ€™s readyState attribute has the value HAVE_FUTURE_DATA or HAVE_ENOUGH_DATA, then the user agent must run the media session invocation algorithm, passing in the media element as current media element. The media session invocation algorithm takes one argument, current media element, and consists of the following steps:

  1. Let media session be the value of current media elementâ€™s current media session.
  2. If media session is empty or media sessionâ€™s current state is ACTIVE abort the remaining steps and return.
  3. Optionally, based on platform conventions, request the most appropriate platform-level media focus for media session based on its current intrinsic media category state.

    If this step is run and platform-level media focus can not be obtained for any reason then pause current media element, abort the remaining steps and return.

  4. Run these substeps:

    1. Optionally, based on platform conventions, the user agent may show an ongoing media interface in the underlying platformâ€™s notifications area and/or show an ongoing media interface in the underlying platformâ€™s lock screen area for media session.
    2. Optionally, based on platform conventions, the user agent may allow hardware and/or software media keys to control playback of media sessionâ€™s active participating media elements.
    If media sessionâ€™s intrinsic media category state is Transient or Transient Solo
    The user agent should not provide user access to any hardware or software media keys and not display any ongoing media interface in the underlying platformâ€™s notifications area or show any ongoing media interface in the underlying platformâ€™s lock screen area for Transient or Transient Solo-based media sessions.

    If the current media elementâ€™s top-level browsing context is not at least partially visible then pause current media element, abort the remaining steps and return.

    The user agent should not provide user access to any hardware or software media keys and not display any ongoing media interface in the underlying platformâ€™s notifications area or show any ongoing media interface in the underlying platformâ€™s lock screen area for Ambient-based media sessions.

    Otherwise, media sessionâ€™s intrinsic media category state is Default

8. Media session interruption

A media session may be interrupted at any time after its media session invocation algorithm has been run. This typically occurs when another application or media session requests and is granted platform-level media focus.

Interruptions can be both transient or permanent. A transient interruption means we can either choose to duck our content and continue playing it out at a reduced volume for the duration of the interruption; or pause our content for the duration of the interruption and then resume playback when the interruption ends. A permanent interruption means we must pause our media content indefinitely.

When a media session receives a start-of-interruption notification from the platform then the user agent must run the media session interruption algorithm passing in the media session as interrupted media session.

The media session interruption algorithm takes one argument, interrupted media session, and consists of running the following steps.

  1. If interrupted media sessionâ€™s current state is not ACTIVE abort the remaining steps and return.
  2. Let interrupting media category be the media category that triggered this interruption. If this interruption has no known media category, let interrupting media category be Default.
  3. Run these substeps:
    If interrupting media category is Content
    1. If interrupted media sessionâ€™s intrinsic media category state is Default or Content then pause all of interrupted media sessionâ€™s active participating media elements and set interrupted media sessionâ€™s current state to IDLE.

      This implies that the next time the media session invocation algorithm is run against a media element in media session that we will re-request media focus according to the steps defined in that algorithm.

    2. Optionally, based on platform conventions, the user agent must remove any previously established ongoing media interface in the underlying platformâ€™s notifications area and any ongoing media interface in the underlying platformâ€™s lock screen area for interrupted media session, if any.
    3. Optionally, based on platform conventions, the user agent must prevent any hardware and/or software media keys from controlling playback of interrupted media sessionâ€™s active participating media elements.
    If interrupting media category is Transient
    1. If interrupted media sessionâ€™s intrinsic media category state is Default or Content then duck all of interrupted media sessionâ€™s active participating media elements and set interrupted media sessionâ€™s current state to INTERRUPTED.

      This implies that the next time the media session release algorithm is run that we will re-activate this media session according to the steps defined in that algorithm.

    If interrupting media category is Transient Solo
    1. If interrupted media sessionâ€™s intrinsic media category state is Default, Content, Transient or Transient Solo then pause all of interrupted media sessionâ€™s active participating media elements and set current media sessionâ€™s current state to INTERRUPTED.

      This implies that the next time the media session invocation algorithm is run against a media element in media session that we will re-request media focus according to the steps defined in that algorithm.

      It also implies that the next time the media session release algorithm is run that we will re-activate this media session according to the steps defined in that algorithm.

When a media session receives an end-of-interruption notification from the platform then the user agent must run the media session continuation algorithm against all known media sessions, passing in each media session as incumbent media session.

The media session continuation algorithm takes one argument, incumbent media session, and consists of running the following steps.

  1. If incumbent media sessionâ€™s current state is not INTERRUPTED then abort the remaining steps and return.
  2. Let interrupting media category be the media category that intially triggered this interruption. If this interruption has no known media category, let interrupting media category be Default.
  3. Run these substeps:
    If interrupting media category is Transient
    1. If incumbent media sessionâ€™s intrinsic media category state is Default or Content then unduck all of incumbent media sessionâ€™s active participating media elements and set incumbent media sessionâ€™s current state to ACTIVE.
    If interrupting media category is Transient Solo
    1. If incumbent media sessionâ€™s intrinsic media category state is Default, Content, Transient, or Transient Solo then unpause the incumbent media sessionâ€™s participating media elements that were active prior to the corresponding start-of-interruption notification and set incumbent media sessionâ€™s current state to ACTIVE.

9. Media session release

When a media elementâ€™s playback has ended, the user agent must remove the media element from its current media sessionâ€™s active participating media elements and then run the media session release algorithm passing in the media elementâ€™s current media session, if any, as the only argument.

When a media element with an intrinsic media category state set to Transient or Transient Solo is paused, then the user agent must remove the media element from its current media sessionâ€™s active participating media elements and then run the media session release algorithm passing in the media elementâ€™s current media session as the only argument.

Any time a media elementâ€™s load algorithm is run by the user agent, it must remove the media element from its current media sessionâ€™s active participating media elements and then also run the media session release algorithm passing in the media elementâ€™s current media session, if any, as the only argument.

The media session release algorithm takes one argument, current media session, and consists of running the following steps.

  1. If current media session is empty or its current state is IDLE then abort the remaining steps and return.
  2. If current media session has one or more active participating media elements then abort the remaining steps and return.
  3. Optionally, based on platform conventions, the user agent must release any currently held platform media focus for current media session.
  4. Optionally, based on platform conventions, the user agent must remove any previously established ongoing media interface in the underlying platformâ€™s notifications area and any ongoing media interface in the underlying platformâ€™s lock screen area for current media session, if any.
  5. Optionally, based on platform conventions, the user agent must prevent any hardware and/or software media keys from controlling playback of current media sessionâ€™s active participating media elements.
  6. Set current media sessionâ€™s current state to IDLE.

    This implies that the next time the media session invocation algorithm is run against any media element in current media session that we will re-request media focus according to the steps defined in that algorithm.

When a media element is removed from a Document and is not attached to another Document, remove the media element from its current media sessionâ€™s active participating media elements and then run the media session release algorithm passing in the media elementâ€™s current media session, if any, as the only argument.
